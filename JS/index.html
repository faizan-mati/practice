üìäPart 1: Data Types & Type Coercion

üî¢ JavaScript Data Types

JavaScript has 8 main data types. Let's see them in action:

// Primitive Types let myString = "Hello World"; // String let myNumber = 42; // Number let myBoolean = true; // Boolean let myUndefined; // Undefined (no value assigned) let myNull = null; // Null (intentionally empty) let mySymbol = Symbol("id"); // Symbol (unique identifier) let myBigInt = 123456789n; // BigInt (large numbers) // Non-Primitive Type let myObject = { name: "John" }; // Object // Check types using typeof console.log(typeof myString); // "string" console.log(typeof myNumber); // "number" console.log(typeof myBoolean); // "boolean" console.log(typeof myUndefined); // "undefined" console.log(typeof myNull); // "object" (this is a famous JS quirk!) console.log(typeof mySymbol); // "symbol" console.log(typeof myBigInt); // "bigint" console.log(typeof myObject); // "object"
‚ö†Ô∏è Important: typeof null returns "object" - this is a known bug in JavaScript that's kept for backward compatibility!

ü§î Understanding NaN (Not a Number)

NaN is a special numeric value that represents "Not a Number":

let result1 = "hello" * 2; // NaN let result2 = Math.sqrt(-1); // NaN let result3 = 0 / 0; // NaN console.log(result1); // NaN console.log(typeof result1); // "number" (weird, right?) // Checking for NaN console.log(isNaN(result1)); // true console.log(Number.isNaN(result1)); // true (preferred method) // NaN is not equal to itself! console.log(NaN === NaN); // false console.log(NaN == NaN); // false
üí° Pro Tip: Use Number.isNaN() instead of isNaN() for more accurate NaN checking!

‚öñÔ∏è == vs === (The Great Equality Debate!)

This is one of the most important concepts in JavaScript:

== (Loose Equality) === (Strict Equality)
Converts types before comparing No type conversion
5 == "5" ‚Üí true 5 === "5" ‚Üí false
Can be unpredictable Always predictable
// == (Loose Equality) - Type Coercion happens console.log(5 == "5"); // true (string "5" converted to number) console.log(true == 1); // true (true converted to 1) console.log(false == 0); // true (false converted to 0) console.log(null == undefined); // true (special case) console.log("" == 0); // true (empty string converted to 0) // === (Strict Equality) - No type conversion console.log(5 === "5"); // false (different types) console.log(true === 1); // false (different types) console.log(false === 0); // false (different types) console.log(null === undefined); // false (different types) console.log("" === 0); // false (different types) // Same type comparisons work the same way console.log(5 === 5); // true console.log("hello" === "hello"); // true
üéØ Best Practice: Always use === and !== unless you specifically need type coercion. It makes your code more predictable and easier to debug!

üîßPart 2: Functions & Scope

üì¶ Function Declaration vs Function Expression

// Function Declaration (Hoisted) function sayHello(name) { return "Hello, " + name + "!"; } // Function Expression (Not Hoisted) const sayGoodbye = function(name) { return "Goodbye, " + name + "!"; }; // Arrow Function (ES6+) const sayHi = (name) => { return "Hi, " + name + "!"; }; // Arrow Function (Shorter syntax) const greet = name => `Hey, ${name}!`; console.log(sayHello("Alice")); // "Hello, Alice!" console.log(sayGoodbye("Bob")); // "Goodbye, Bob!" console.log(sayHi("Charlie")); // "Hi, Charlie!" console.log(greet("Diana")); // "Hey, Diana!"

üéà Hoisting (The Magic Elevator!)

Hoisting means declarations are "moved" to the top of their scope:

// This works because of hoisting! console.log(hoistedFunction()); // "I'm hoisted!" function hoistedFunction() { return "I'm hoisted!"; } // But this doesn't work: try { console.log(notHoisted()); // TypeError: notHoisted is not a function } catch(e) { console.log("Error:", e.message); } var notHoisted = function() { return "I'm not hoisted!"; }; // Variables are hoisted but not their values console.log(myVar); // undefined (not an error!) var myVar = "Hello"; console.log(myVar); // "Hello" // let and const are hoisted but in "temporal dead zone" // console.log(myLet); // ReferenceError let myLet = "World";

üîí Scope (Variable Accessibility)

// Global Scope var globalVar = "I'm global!"; let globalLet = "I'm also global!"; function scopeDemo() { // Function Scope var functionVar = "I'm in function scope!"; let functionLet = "Me too!"; if (true) { // Block Scope var blockVar = "I'm var in block"; // Still function scoped! let blockLet = "I'm let in block"; // Block scoped const blockConst = "I'm const in block"; // Block scoped console.log(blockVar); // Works console.log(blockLet); // Works console.log(blockConst); // Works } console.log(blockVar); // Works (var ignores block scope) // console.log(blockLet); // ReferenceError // console.log(blockConst); // ReferenceError console.log(globalVar); // Works console.log(functionVar); // Works } scopeDemo();

üîê Closures (Functions Remember!)

A closure gives you access to an outer function's scope from an inner function:

// Simple Closure Example function outerFunction(x) { // This is the outer function's variable function innerFunction(y) { // Inner function has access to outer function's variables return x + y; // x is from outer scope! } return innerFunction; } const addFive = outerFunction(5); console.log(addFive(3)); // 8 (5 + 3) // Practical Closure Example - Counter function createCounter() { let count = 0; // Private variable return function() { count++; // Inner function can modify outer variable return count; }; } const counter1 = createCounter(); const counter2 = createCounter(); console.log(counter1()); // 1 console.log(counter1()); // 2 console.log(counter2()); // 1 (separate counter!) console.log(counter1()); // 3

‚ö° IIFE (Immediately Invoked Function Expression)

Functions that run immediately and create their own scope:

// Basic IIFE Syntax (function() { console.log("I run immediately!"); })(); // IIFE with parameters (function(name) { console.log("Hello, " + name + "!"); })("JavaScript"); // IIFE with return value const result = (function(a, b) { return a + b; })(10, 20); console.log(result); // 30 // Arrow function IIFE (() => { console.log("Arrow IIFE!"); })(); // Why use IIFE? To avoid polluting global scope (function() { var privateVar = "I won't pollute global scope!"; // This variable is contained within IIFE })(); // console.log(privateVar); // ReferenceError

üèπ Arrow Functions (ES6 Magic)

// Regular function function regularAdd(a, b) { return a + b; } // Arrow function equivalents const arrowAdd1 = (a, b) => { return a + b; }; const arrowAdd2 = (a, b) => a + b; // Implicit return const square = x => x * x; // Single parameter, no parentheses needed const sayHello = () => "Hello World!"; // No parameters // Arrow functions in arrays const numbers = [1, 2, 3, 4, 5]; // Using map with arrow function const doubled = numbers.map(num => num * 2); console.log(doubled); // [2, 4, 6, 8, 10] // Using filter with arrow function const evenNumbers = numbers.filter(num => num % 2 === 0); console.log(evenNumbers); // [2, 4]
‚ö†Ô∏è Important: Arrow functions don't have their own 'this' context - they inherit it from the parent scope. This is great for callbacks but different from regular functions!
